1. SYSTEM ARCHITECTURE
================================================================================

1.1 HIGH-LEVEL OVERVIEW
-----------------------
ReleaseHub is a backend API + frontend app that answers natural-language
questions about OS versions and Linux patches by querying the ReleaseTrain
external API. The backend parses the user question (vendor, date, intent),
routes to either "patch" or "OS" flow, fetches data from ReleaseTrain, picks
the matching document, and returns a structured answer.

  [User Question] → [Backend: server.js]
                         ↓
              Parse: vendor, date, "patch" vs "OS"
                         ↓
         ┌───────────────┴───────────────┐
         ↓                               ↓
   PATCH FLOW                      OS FLOW
   (Linux patch only)              (Android, iOS, Windows, linux-dist, etc.)
         ↓                               ↓
   fetchPatchSearch(vendor)        fetchOSComponents()
   findLinuxPatchByDate()          findOSByProductAndDate(components, product, date)
   docHasVendor(doc, vendor)      formatOSResponse(doc)
   formatLinuxPatchResponse()              ↓
         ↓                               Response
   Response

1.2 BACKEND (Node.js)
---------------------
- Entry: src/server.js (Express app)
- Port: process.env.PORT || 3000
- Key modules:
  - releasetrain.js: fetchOSComponents(), fetchPatchSearch(), findOSByProductAndDate(),
    findLinuxPatchByDate(), formatLinuxPatchResponse(), docHasVendor()
  - schema-os.js: formatOSResponse(), OS_COMPONENT_SCHEMA

Request flow for POST /answer:
  1. Read question from body (question | prompt | q).
  2. pushLastPrompt(question) → Redis list (if REDIS_URL set).
  3. parsePatchVendor(question) → "patch for X" → X.
  4. If X is non-Linux patch vendor → return "I don't know...".
  5. If patch for Linux or question contains "linux" (and not "android") → PATCH FLOW.
  6. Else → OS FLOW: getVendorNames(), resolveVendorFromQuestion(), findOSByProductAndDate().
  7. Format and return JSON (answer, version, main, additional, vendor, source).

Date parsing: DD-MM-YYYY or DD/MM/YYYY anywhere in question → YYYYMMDD.

1.3 FRONTEND (React + Vite)
---------------------------
- Location: frontend/
- Dev: npm run dev (Vite dev server)
- Build: npm run build
- Sends POST /answer with { question: "..." } (or prompt / q).
- Displays answer, version, product, release notes, license; optional debug panel.

1.4 EXTERNAL DATA SOURCES
-------------------------
- OS components: https://releasetrain.io/api/component?q=os
- Patch search:  https://releasetrain.io/api/v/search?q=<vendor>&channel=patch&limit=25&page=1
- Vendor names:  https://releasetrain.io/api/c/names

1.5 OPTIONAL: REDIS
-------------------
- If REDIS_URL is set, last N prompts are stored in Redis (releasehub:last_prompts).
- Used for history/debugging; not required for core answer flow.
